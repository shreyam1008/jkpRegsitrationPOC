# Coding Standards and Technical Guidelines

This document outlines the strict coding standards and architectural boundaries for the JKP Registration system. Both human developers and AI assistants must adhere to these guidelines to ensure a robust, secure, and maintainable codebase.

---

## 1. Global Principles

*   **Readability Over Cleverness:** Write code that is easy to understand. Avoid highly terse, "clever" one-liners if they sacrifice readability. 
*   **Explicit is Better than Implicit:** Never rely on implicit type inference at system boundaries (e.g., function arguments, API returns).
*   **Dependency Minimization:** Do not introduce new third-party libraries unless absolutely necessary. Every dependency is a potential security risk and maintenance burden.
*   **Error Handling:** Never swallow exceptions silently. Log errors with context (using Sentry in production) and return user-friendly, generalized error messages to the client.

### 2.1 Core Stack
*   **Language:** Strict TypeScript(6.0)
*   **Framework:** React 19.2+
*   **Styling:** TailwindCSS (4.2)

*   **Virtualization:** TanStack Virtual (for rendering large lists smoothly)
*   **Package Manager:** Bun

*   **Forms:** React Hook Form
*   **State Management:** React Context
*   **Animations:** Native CSS/JS (GPU accelerated)
*   **Formatter/Linter:** oxfmt / oxlint 
*   **Dev Server:** Vite



### 2.2 TypeScript Rules
*   **No `any`:** The use of `any` is strictly prohibited. If a type is truly unknown, use `unknown` and perform runtime type narrowing.
*   **Explicit Interfaces:** Define explicit `interface` or `type` declarations for all component props, React Context state, and API responses.

### 2.3 State Management & Data Fetching
*   **Server State:** Use `@tanstack/react-query` for all server communication. This handles browser-side caching, loading states, error states, and automatic retries.
*   **Local UI State:** Use standard React Context or `useState` for transient UI state (e.g., "is a modal open?"). Avoid heavy global state managers (like Redux) unless the state becomes extremely complex.

### 2.4 Communication Layer
*   **REST via TanStack Query:** All backend communication must use `@tanstack/react-query` with typed fetch wrappers against the REST API. No raw `fetch()` or `axios` calls scattered across components.
*   **Typed API Layer:** Define a centralized API client module (e.g., `src/api/`) with typed request/response interfaces for each endpoint. Components consume these via TanStack Query hooks.
*   **Pre-Signed URLs:** The only exception is media uploads, which should use native `fetch`/`XMLHttpRequest` to `PUT` files directly to SeaweedFS using pre-signed URLs generated by the backend.

### 2.5 Structure
*   **Feature-First Organization:** Organize the `src` directory by feature domain (e.g., `src/features/satsangi-registration/`, `src/features/dashboard/`) rather than flat technical directories (e.g., putting all components in one folder, all hooks in another).

## 3. Backend Standards (C# / .NET)

The backend prioritizes compile-time type-safety, performance, and strict separation of concerns.

### 3.1 Core Stack
*   **Language:** C# 12+ (.NET 8 LTS or later)
*   **Framework:** ASP.NET Core (REST API via Minimal APIs or Controllers, with optional gRPC for perf-critical endpoints)
*   **API Documentation:** Swagger/OpenAPI auto-generated from controllers/DTOs.
*   **gRPC (Optional):** `grpc-dotnet` available for bulk data transfer, server-to-server calls, or streaming. Can coexist with REST in the same project.
*   **ORM:** Entity Framework Core (code-first migrations, LINQ queries). Raw SQL/Dapper for bulk ETL or complex reporting only.
*   **Database:** PostgreSQL (accessed via `Npgsql` EF Core provider)
*   **Authentication:** ASP.NET Identity (Phase 1). SSO via OpenIddict or Keycloak in Phase 2 — see `008-dotnet-architectural-choices.md` §7.
*   **Background Tasks:** `BackgroundService` + `Channel<T>` for in-process queuing. Hangfire (PostgreSQL-backed) for durable/scheduled tasks.
*   **Caching:** `IMemoryCache` for in-process static data. `IDistributedCache` interface for future Redis swap.
*   **Object Storage:** SeaweedFS (S3-compatible, Apache 2.0) via `AWSSDK.S3`. Pre-signed URLs for browser-direct uploads.

### 3.2 C# Rules
*   **Nullable Reference Types:** Must be enabled project-wide (`<Nullable>enable</Nullable>`). All nullable references must be explicitly marked with `?`.
*   **No `dynamic`:** The use of `dynamic` is strictly prohibited. If a type is truly unknown, use generics or explicit interfaces.
*   **Explicit Types at Boundaries:** All API DTOs, service method parameters, return types, and public surfaces must use explicit types. `var` is permitted only for local variables where the type is obvious from the right-hand side.
*   **Formatting:** Code must be auto-formatted using `dotnet format`. Analyzers (e.g., `Microsoft.CodeAnalysis.NetAnalyzers`) must be enabled with warnings-as-errors in CI/CD.
*   **Async All the Way:** All I/O operations (database, file, network) must use `async`/`await`. Never call `.Result` or `.Wait()` on tasks — this causes deadlocks in ASP.NET Core.

### 3.3 Architectural Boundaries
*   **Service Layer:** Controllers/endpoints must be thin — they validate input, call the service/repository layer, and map results to DTOs. Core business logic belongs in injectable service classes, not in controller methods.
*   **Dependency Injection:** All services, repositories, and infrastructure components must be registered in the DI container. Never use `new` to instantiate services with dependencies.
*   **No Business Logic in Controllers/Endpoints:** Controllers handle HTTP concerns (routing, model binding, status codes). Business rules belong in the service layer.
*   **gRPC Separation (if used):** Keep `protobuf` generated files (from `Grpc.Tools`) strictly isolated in a `Protos/` or `Generated/` directory. gRPC services must delegate to the same service layer as REST controllers — no duplicated logic.

### 3.4 Data & Concurrency
*   **Streaming:** Avoid loading massive datasets (e.g., 50,000 rows) directly into RAM. Use `IAsyncEnumerable<T>` and EF Core's `AsAsyncEnumerable()` to stream data.
*   **Connection Pooling:** EF Core manages connection pooling via `Npgsql`. Do not create raw `NpgsqlConnection` instances outside of the DI-managed `DbContext`.
*   **Background Tasks:** Any task that takes longer than 1-2 seconds (e.g., ETL scripts, CSV exports, bulk notifications, image compression) MUST be offloaded to a `BackgroundService` via `Channel<T>`. The API handler must return immediately after enqueuing the task.
