# Coding Standards and Technical Guidelines

This document outlines the strict coding standards and architectural boundaries for the JKP Registration system. Both human developers and AI assistants must adhere to these guidelines to ensure a robust, secure, and maintainable codebase.

---

## 1. Global Principles

*   **Readability Over Cleverness:** Write code that is easy to understand. Avoid highly terse, "clever" one-liners if they sacrifice readability. 
*   **Explicit is Better than Implicit:** Never rely on implicit type inference at system boundaries (e.g., function arguments, API returns).
*   **Dependency Minimization:** Do not introduce new third-party libraries unless absolutely necessary. Every dependency is a potential security risk and maintenance burden.
*   **Error Handling:** Never swallow exceptions silently. Log errors with context (using Sentry in production) and return user-friendly, generalized error messages to the client.

The frontend is a single-page application focused on high-speed data entry and validation.

### 2.1 Core Stack
*   **Language:** Strict TypeScript(6.0)
*   **Framework:** React 19.2+
*   **Styling:** TailwindCSS (4.2)

*   **Package Manager:** Bun

*   **State Management:** React Context
*   **Animations:** Native CSS/JS (GPU accelerated)
*   **Formatter/Linter:** oxfmt / oxlint 
*   **Dev Server:** Vite



### 2.2 TypeScript Rules
*   **No `any`:** The use of `any` is strictly prohibited. If a type is truly unknown, use `unknown` and perform runtime type narrowing.
*   **Explicit Interfaces:** Define explicit `interface` or `type` declarations for all component props, React Context state, and gRPC responses.

### 2.3 State Management & Data Fetching
*   **Server State:** Use `@tanstack/react-query` for all server communication. This handles browser-side caching, loading states, error states, and automatic retries.
*   **Local UI State:** Use standard React Context or `useState` for transient UI state (e.g., "is a modal open?"). Avoid heavy global state managers (like Redux) unless the state becomes extremely complex.

### 2.4 Communication Layer
*   **gRPC-Web Only:** All backend communication must flow through the generated `grpc-web` clients. 
*   **No Manual Fetches:** Do not write manual `fetch()` or `axios` calls to the gRPC backend. 
*   **Pre-Signed URLs:** The only exception to the gRPC rule is media uploads, which should use native `fetch`/`XMLHttpRequest` to `PUT` files directly to MinIO using pre-signed URLs generated by the backend.

### 2.5 Structure
*   **Feature-First Organization:** Organize the `src` directory by feature domain (e.g., `src/features/satsangi-registration/`, `src/features/dashboard/`) rather than flat technical directories (e.g., putting all components in one folder, all hooks in another).

## 3. Backend Standards (Python / gRPC)

The backend prioritizes type-safety, performance, and strict separation of concerns.

### 3.1 Core Stack
*   **Language:** Python 3.10+
*   **Primary API Framework:** `grpcio` (Native gRPC)
*   **Proxy/HTTP Framework:** `FastAPI` (Strictly limited to the grpc-web proxy and specific HTTP-only endpoints like Webhooks, and REST be used by fastapi only in need basis like exposing api to other apps)
*   **Database:** PostgreSQL (accessed via SQLAlchemy or raw async drivers)

### 3.2 Python Rules
*   **Strict Type Hinting:** Mandatory for all function arguments and return types. Use the `typing` module extensively.
*   **Formatting:** Code must be auto-formatted and linted using `ruff` (format + lint + import sorting), and type-checked in CI/CD using `mypy`.

### 3.3 Architectural Boundaries
*   **Separation of Protobuf:** Keep `protobuf` generated files (`_pb2.py`, `_pb2_grpc.py`) strictly isolated in an `api/` or `generated/` directory. Do not mix business logic into these generated files.
*   **FastAPI Restrictions:** Do not put core business logic in FastAPI routes. FastAPI's primary job is to act as the reverse proxy for `grpc-web` or handle external Webhooks. Core logic belongs in native gRPC servicers.

### 3.4 Data & Concurrency
*   **Streaming & Generators:** Avoid loading massive datasets (e.g., 50,000 rows) directly into RAM. Use database cursors and Python generators (`yield`) to stream data.
*   **Connection Pooling:** All database interactions must use a robust connection pool (e.g., `psycopg v3`, `PgBouncer`, or SQLAlchemy's built-in pool) to prevent connection exhaustion.
*   **Background Tasks:** Any task that takes longer than 1-2 seconds (e.g., ETL scripts, CSV exports, bulk notifications, image compression, document encryption) MUST be decoupled from the gRPC request-response cycle. Push the task to the another threda asyncio.to_thread() or PostgreSQL-backed Background Queue.
